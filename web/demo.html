VAD 状态机

// server/asr.js:224-257 — VAD 状态机
if (this.vadState === 'silence') {
  if (isSpeech) {
    this.vadSpeechSamples += samples;
    if (this.vadSpeechSamples >= this.vadSpeechOnset) {
      this.vadState = 'speech';
    }
  }
} else {  // vadState === 'speech'
  if (!isSpeech) {
    this.vadSilenceSamples += samples;
    if (this.vadSilenceSamples >= this.vadEndpointSamples) {
      this.vadState = 'silence';
      this._processChunk();
      this._flushAccumulated();
    }
  }
}

LocalAgreement-2 提交策略

连续两次 ASR 假设在前缀达成一致才定稿，平衡速度与稳定性。源自 whisper_streaming 的经典做法。
normalize: lowercase + 去标点
flush 触发: final 事件 / VAD 端点
hypothesist-1: The quick brown fox jumps
hypothesist  : The quick brown fox runs

commit: "The quick brown"  |  pending: "fox runs" 
// server/commit.js:20-41 — LocalAgreement-2
update(words) {
  const newlyCommitted = [];
  const start = this.committed.length;
  let i = start;
  while (
    i < words.length &&
    i < this.prevHypothesis.length &&
    normalize(words[i]) === normalize(this.prevHypothesis[i])
  ) {
    newlyCommitted.push(words[i]);
    i++;
  }
  this.committed.push(...newlyCommitted);
  this.prevHypothesis = words;
  return { committed, pending, newlyCommitted };
}

Step 1 — 浏览器端采集
  麦克风前 5s PCM16 → collectVoiceSample() 累积到 80000 样本
  UI 显示「采集声纹 xx%」进度，TTS 播放期间暂停采集防止回声
Step 2 — 发送到服务端
  先发 JSON: { type: 'voice_sample', sampleRate: 16000, samples: 80000 }
  紧接着发 二进制 ArrayBuffer（160KB PCM16）
  服务端用 pendingVoiceSample 标志位区分声纹 vs 普通音频
Step 3 — 服务端处理
  PCM16 → pcm16ToWav() 手动拼 44 字节 WAV 头 → Base64 → data URI
  WAV → transcribeWav() 调 SenseVoice ASR 得到文本 transcript
  存入 sessionVoiceRefs Map（会话级，断开自动清理）
Step 4 — 克隆合成
  翻译完成 → synthesize(text, sessionId)
  长文本按标点拆分（/[。！？；，、.!?;,]/），只合成第一段降低延迟
  有 voiceRef → references 数组克隆；无 → 预设音色 fallback
Step 5 — 浏览器播放
  服务端发 tts_audio JSON + MP3 二进制 → Blob → new Audio() 播放
  播放期间暂停 Web Speech API + 静音麦克风 PCM 防止回声循环

自动纠正机制

四层纠正管线 — 从语音识别到人工校对，层层保证译文质量

Layer 1 — ASR 级

语音假设逐步稳定

Web Speech API 逐词输出 interim hypothesis，文字不断跳动修正。 LocalAgreement-2 算法：只有连续两次假设在前缀达成一致才定稿提交，其余留作灰色临时区。

interim: "I think we should" → interim: "I think we should go" → commit: "I think we should"

服务端 ASR（SenseVoice/Whisper）同理：VAD 分块 → 整段识别 → 文字累积 → 句边界检测后 flush 定稿。

Layer 2 — LLM 翻译回改

上下文感知回溯修正

8 句滑动窗口作为上下文注入 Prompt。LLM 翻译当前句的同时，输出 corrections[] 数组回溯修改之前已发布的译文。 典型场景：后文出现消歧信息后，LLM 自动修正前句的错误翻译。

// LLM 输出 JSON 结构
{
  "target": "当前句的译文",
  "corrections": [
    {"id": 3, "target": "修正后的第3句译文"},
    {"id": 5, "target": "修正后的第5句译文"}
  ],
  "terms": [{"src": "术语", "tgt": "翻译"}]
}

系统遍历 corrections 数组 → 找到对应 id 的字幕行 → 原地替换译文 → 触发 Layer 3 视觉反馈。

Layer 3 — 字幕三态显示

视觉状态即时反馈

每行字幕都有明确的视觉状态，用户一眼就能分辨定稿、临时和修正内容：

白色 = 已定稿 灰色 = 临时 (is-partial) 黄色闪烁 = 已修正

修正动画：flash-correct CSS 关键帧 1.6s，黄色高亮 + 右上角"已修正"标签。动画结束后自动移除 class。

Layer 4 — 人工修正

双击编辑 → 即时重译

用户双击任意原文字幕行，进入 contentEditable 编辑模式。修改完成后自动发送重译请求：

双击编辑原文 → ws.send({type:'retranslate', id, text}) → 服务端 LLM 重译 → 译文自动更新

修改后的原文同步更新翻译上下文窗口（history[]），影响后续句子的翻译质量和术语一致性。

实时 ASR 路径

全部走服务端 Whisper

翻译引擎

LLM (DeepSeek)
Google Translate API专用 NMT
LLM 能理解上下文（8 句滑动窗口）、做术语一致性（自动术语表）、回溯纠正已发译文（corrections 机制），这些传统 NMT 做不到。流式输出搭配 Regex 提取实现边生成边显示。

VAD 方案

自实现能量 VAD
Silero VADWebRTC VAD
30 行代码实现双阈值状态机，RMS 能量检测足够可靠。避免加载 ONNX runtime（Silero）或额外 WASM（WebRTC VAD）。参数可按场景调整：实时模式更灵敏（0.6s 端点），文件模式更宽松（1.0s）。

直接显示 interim等 final
直接显示 interim 字幕疯狂跳动；等 final 太慢（一整句才出来）。LA-2 是 whisper_streaming 等流式同传系统的经典做法：只有连续两次假设在前缀一致才定稿，平衡实时性和稳定性。

5 种输入模式

每种模式的完整数据流路径和实现细节

🎤

模式一：麦克风实时

最低延迟
getUserMedia() → AudioContext → ScriptProcessor (4096) → 降采样 → PCM16@16kHz → WebSocket 二进制帧

音频采集：浏览器 getUserMedia 获取麦克风，开启 echoCancellation + noiseSuppression + autoGainControl。 ScriptProcessor 每 4096 样本触发一次回调，从浏览器原生采样率（通常 44.1kHz/48kHz）降采样为 16kHz PCM16。

双路 ASR：浏览器端同时启动 Web Speech API（逐词 interim results → LocalAgreement-2 稳定字幕）， 音频二进制帧同时发往服务端 SenseVoice/Whisper（VAD 分块 → 整段识别）。两路结果互补。

声音克隆采样：前 5 秒音频（80000 samples）自动收集，发送给服务端做 ASR 转录 + CosyVoice2 声纹注册。 TTS 播放期间暂停采集，防止回声被录入。

📄

模式二：文件上传

音频 / 视频
File → POST /upload → FFmpeg 转码 PCM16 → 返回 ArrayBuffer → 0.5s 分块发送 → 服务端 ASR

服务端转码：文件通过 HTTP POST 上传到 /upload 端点，服务端用 FFmpeg 转为 16kHz 单声道 PCM16 并返回整段 ArrayBuffer。

模拟实时流：客户端将 PCM 切成 8000 samples（0.5 秒）的块，每隔 100ms 发送一块到 WebSocket。 服务端 StreamingASR 按照与实时模式相同的 VAD 逻辑处理（但端点检测更宽松：1.0s）。进度条实时更新百分比。

视频同步：如果上传的是视频文件，自动创建 <video> 预览，ASR 的每句结果记录时间戳。 播放视频时通过 ontimeupdate 高亮当前时间对应的字幕行，其余行淡出。翻译完成后可一键导出 SRT。

声纹提取：自动截取文件前 5 秒 PCM 作为声音克隆参考，与分块音频分开发送（先发 voice_sample 消息头 + 二进制数据）。

🎬

模式三：标签页音频捕获

画中画字幕
getDisplayMedia() → 提取 audioTrack → 停掉 videoTrack → ScriptProcessor (4096) → 降采样 → WS

屏幕共享 API：调用 getDisplayMedia({ video: true, audio: true })，用户选择标签页并勾选「分享音频」。 获取 MediaStream 后立即 停止视频轨道（只要音频），节省资源。监听音频轨道的 ended 事件，用户从浏览器 UI 停止共享时自动清理。

画中画悬浮窗：在 getDisplayMedia 之前（需要 user gesture）调用 Document Picture-in-Picture API 创建 500x180 悬浮窗口，注入自定义 CSS + DOM。窗口始终置顶，显示原文 + 译文两行字幕，实时更新。 不支持 Document PiP 的浏览器降级为 window.open 弹窗。

音频处理：与麦克风模式共用同一套 AudioContext → ScriptProcessor → 降采样 → WebSocket 管线。 区别是不启动浏览器 Web Speech API（标签页音频无法用浏览器 ASR），全部依赖服务端 ASR。

🌐

模式四：在线视频 URL

YouTube / 直播
URL → WebSocket → yt-dlp -f bestaudio → stdout | pipe → FFmpeg → PCM16@16kHz → 服务端 ASR

全服务端处理：客户端只发送一个 start_url JSON 消息，所有重活在服务端完成。

管道架构：服务端 spawn 两个子进程：yt-dlp 用 -f bestaudio -o - 提取最佳音频流输出到 stdout； FFmpeg 从 pipe:0 读取，转码为 pcm_s16le -ar 16000 -ac 1 输出到 pipe:1。 两个进程通过 ytdlp.stdout.pipe(ff.stdin) 串联，实现流式处理——不需要先下载完整文件。

实时识别：FFmpeg 的 stdout 产出 PCM 流，直接喂入 StreamingASR 的 feedChunk()， 与麦克风/文件模式共用同一套 VAD + ASR + 翻译管线。支持 YouTube、Bilibili 等 yt-dlp 支持的 1000+ 站点。

📺

模式五：OBS 字幕叠加

直播场景
overlay.html?mode=both&dir=en2zh → WebSocket (只读) → 接收 asr_final / translation → 渲染字幕 DOM

独立页面：OBS 叠加使用独立的 overlay.html，背景完全透明（background: transparent）， 字幕居下居中显示，带 text-shadow 增加可读性。原文 24px 半透明 + 译文 32px 粗体，淡入动画。

只读客户端：overlay 连接同一个 WebSocket 服务端，发送 mode: 'overlay' 标记自己是只读客户端。 服务端不为 overlay 创建 ASR 实例，只广播其他活跃会话产生的识别和翻译结果。

使用方式：主应用界面一键复制 OBS 浏览器源 URL（含翻译方向参数）。在 OBS 中添加「浏览器源」，粘贴 URL， 设置宽高后即可叠加到直播画面。支持 mode=source（仅原文）/ target（仅译文）/ both（双语）三种显示模式。

附加功能

说话人分离 — 静音 ≥2s 自动切换 Speaker 0/1，CSS 颜色区分 语言自动检测 — Unicode 码点统计（CJK/假名/韩文/Latin），自动修正翻译方向 术语记忆 — LLM 每次翻译提取 terms[]，最多保留 30 条，注入后续 Prompt SRT 导出 — 支持原文/译文/双语三种格式，文件模式含实际时间戳 实时纠正 — LLM 输出 corrections[] 回溯修改之前句子的译文 可编辑字幕 — 双击字幕行可直接修改，修改结果反馈给翻译上下文
